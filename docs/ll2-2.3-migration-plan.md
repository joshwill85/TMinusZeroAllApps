# LL2 2.3 Migration + Enrichment Plan (Phases 1–3)

This document is a **plan only** for:
1) upgrading our LL2 integration from **v2.2.0 → v2.3.0**,
2) ingesting + caching **launch-level granular fields** (especially `timeline`),
3) ingesting **LL2 Event** details (selectively) and tying them to launches.

It is written to minimize redundancy, respect rate limits, and be rollback-safe.

---

## Confirmed facts (from LL2 docs / OpenAPI)

- **LL2 2.3.0 is the latest supported production version**.
  - 2.2.0 is deprecated/unsupported; 2.4.0 is listed as unreleased.
- 2.3 endpoints use **plural + snake_case** paths (e.g. `/launches/`, `/locations/`).
- The “timelineevent” concept we care about is **not a standalone timelineevent endpoint**; it is delivered as:
  - `LaunchDetailed.timeline[] = { type: {id, abbrev, description}, relative_time }`
  - Timeline event types are enumerated via `/config/timeline_event_types/`.
- In 2.3:
  - `infoURLs`/`vidURLs` → `info_urls`/`vid_urls` (and related casing).
  - `holdreason` is **removed** from Launch.
  - Images are **Image objects** (with `credit`, `license`, `thumbnail_url`, `single_use`, etc.).

---

## Goals

- **Upgrade to LL2 2.3.0** without breaking ingestion cadence or UI.
- **Ingest timeline events** (launch milestones / “sub-status”) and expose them in the product.
- Fill and expose other launch-level “2.3 granular” fields that enhance detail pages:
  - `updates`, `mission_patches`, attempt counts, `pad_turnaround`, image credit/licensing.
- Add **event ingestion** to support “events related to a launch” (especially where SNAPI provides LL2 event IDs).
- **Avoid redundant ingestion** (don’t fetch entity endpoints if the launch payload already embeds the needed fields).
- Maintain **rate-limit compliance** and **operational safety** (feature flags, staged rollout, rollback).

---

## Non-goals (for phases 1–3)

- Building a full normalized mirror of all LL2 models (countries, images, orbits, payload catalog, spacecraft, etc.).
- Changing the “US-only” scope (location filter) in this iteration.
- Replacing our internal `launch_updates` diff tracking (that is separate from LL2 `updates`).

---

## Current state (summary)

- LL2 ingestion currently targets **v2.2.0** (`LL2_BASE = https://ll.thespacedevs.com/2.2.0`).
- Launch ingestion uses `mode=detailed` and persists a curated subset into:
  - `public.launches` (paid/live)
  - `public.launches_public_cache` (public read)
- We already have DB columns for some 2.3 launch fields (`timeline`, `updates`, `mission_patches`, image license fields, attempt counts),
  **but we do not populate them** today and do not currently expose them in the UI.

---

## Phase 1 — Upgrade to LL2 2.3.0 (Compatibility-first)

### Objective

Safely switch all LL2 calls from 2.2 to 2.3 and handle breaking changes (path + field rename/removal) without data loss.

### Deliverables

- A single configurable LL2 version (default 2.3.0) across:
  - `supabase/functions/ingestion-cycle`
  - local scripts in `scripts/`
  - any server-side LL2 utilities
- Updated parsing to match 2.3 schema (snake_case fields, image objects).
- “Dark launch” option: ability to run 2.3 ingestion without overwriting critical prod fields until validated.

### Work plan

**A. Decide configuration strategy**

- [ ] Introduce a single source of truth for LL2 base URL (e.g. `LL2_BASE_URL` env or `system_settings.ll2_base_url`).
- [ ] Keep a short-lived fallback plan (ability to revert to 2.2 quickly if needed).

**B. Update endpoints and field mappings**

- [ ] Update launch list endpoint:
  - from `GET /launch/` → `GET /launches/` (still `mode=detailed`).
- [ ] Update locations endpoint used to build the US location filter:
  - from `GET /location/` → `GET /locations/`
  - confirm filters still work (`country_code=USA` still available in 2.3).
- [ ] Update launch field mapping:
  - `infoURLs` → `info_urls`
  - `vidURLs` → `vid_urls`
  - ensure `hold_reason` behavior: 2.2 uses `holdreason`; 2.3 removes it.
    - Plan: keep our DB `hold_reason` column, but accept that it may become null under 2.3 unless another field replaces it.
- [ ] Update provider/agency field mapping:
  - `country_code` becomes `country` object in 2.3; decide what we persist into our `provider_country_code`:
    - Likely map to `country[0].alpha_3_code` or `alpha_2_code` depending on desired format (we currently store `USA`).
- [ ] Update image mapping:
  - from `image` string URLs → `image.thumbnail_url` / `image.image_url` / `image.credit` / `image.license.*` / `image.single_use`.

**C. Rate limit + reliability**

- [ ] Ensure calls still go through our atomic rate limiter (`tryConsumeProvider('ll2')`).
- [ ] Verify `limit=100` paging behavior unchanged.

**D. Validation (pre-rollout)**

- [ ] In a dev/staging environment, fetch a sample set of launches and confirm:
  - launch count unchanged for equivalent filters
  - key fields parse correctly (NET/status/provider/rocket/pad)
  - no ingestion runtime exceptions due to schema mismatch
- [ ] Add temporary logging (or ingestion_run stats) that counts:
  - number of launches processed
  - number of launches where `timeline` exists
  - number of launches where `image` contains license data

**E. Rollout + rollback**

- [ ] Roll out with a feature flag:
  - start with “2.3 fetch + map only” writing to **new columns or JSONB shadow fields** (no UI change yet).
  - then switch primary writes once validated.
- [ ] Rollback plan:
  - toggle back to 2.2 base URL and old mapping if 2.3 breaks ingestion.

### Exit criteria (Phase 1 complete)

- Ingestion runs reliably on 2.3 (no elevated error rate).
- Public cache refresh still succeeds and feed renders.
- We have confirmed which 2.3 fields are available in the launch payload under our filters.

## Phase 2 — Launch-level granular enrichment (timeline, updates, patches, image licensing)

### Objective

Ingest “extra granular” data that is already delivered in `GET /launches/?mode=detailed` (no extra LL2 calls) and expose it to the UI.

**Top priority:** `timeline` (launch milestone events as “sub-status”).

### Deliverables

- Persist the following fields to `public.launches` **and** propagate to `public.launches_public_cache`:
  - `timeline` (array of `{ type: {id, abbrev, description}, relative_time }`)
  - `updates` (LL2 update feed items)
  - `mission_patches`
  - attempt count fields + `pad_turnaround`
  - image licensing fields: `image_url`, `image_credit`, `image_license_name`, `image_license_url`, `image_single_use`
- UI display:
  - Launch detail page displays a “Timeline” section showing upcoming milestones (T- and T+).
  - Optionally, Launch cards show the **next** milestone label when within a configured window (e.g. T-60m → T+10m).

### Work plan

**A. Schema readiness check**

- [ ] Confirm the existing columns added in migration `0025_ll2_23_enrichment.sql` exist in the target DB.
- [ ] Decide whether any new columns are needed for UI convenience:
  - optional `next_timeline_event` (computed) if we want fast queries without recompute
  - optional `timeline_last_computed_at` (if we materialize)
  - default plan: **do not** add computed columns initially; compute in app/UI from `timeline + net`.

**B. Ingestion mapping (launch → DB)**

- [ ] Map and upsert `launch_designator` (2.3 field) into our `launches.launch_designator`.
- [ ] Map attempt count fields and `pad_turnaround`.
- [ ] Map `mission_patches`, `updates`, and `timeline` as JSONB.
  - Keep full fidelity JSON (don’t “simplify” at ingest).
  - Validate JSON size is reasonable for Postgres row limits (should be, but confirm worst case).
- [ ] Map image object:
  - `launch.image.image_url` → `launches.image_url`
  - `launch.image.thumbnail_url` → `launches.image_thumbnail_url`
  - `launch.image.credit` → `launches.image_credit`
  - `launch.image.license.name/link` → `launches.image_license_name/image_license_url`
  - `launch.image.single_use` → `launches.image_single_use`

**C. Public cache propagation**

- [ ] Extend public cache refresh so it copies the new fields into `launches_public_cache`.
  - Ensure these fields are safe to expose publicly (timeline milestones and image licensing should be OK).

**D. UI: timeline display**

- [ ] Define timeline semantics:
  - `relative_time` is an ISO-8601 duration; can be negative or positive.
  - Compute absolute event time: `event_time = net + relative_time`.
  - “Current milestone” = most recent event at or before `now` (within a reasonable window).
  - “Next milestone” = next event strictly after `now`.
- [ ] Launch detail page:
  - display a vertical list of timeline events with:
    - `type.abbrev` (primary label)
    - `type.description` (secondary)
    - time label `T-…` / `T+…` and optionally absolute local time
  - handle missing/empty timelines gracefully.
- [ ] Launch cards (optional, gated behind feature flag):
  - show the next milestone only near launch (avoid noise weeks out).

**E. Historical remediation**

**NOTE:** The one-time historical load is complete; ingestion is incremental + CDC only.
- If a one-time historical remediation is needed, run a targeted, time-bounded repair job (not an always-on job).

**F. Validation**

- [ ] Spot-check a handful of launches and confirm:
  - `timeline` stored and visible in UI
  - computed next/current milestones match LL2 payload
  - image credit/license fields populate correctly
- [ ] Add sanity metrics in ingestion runs:
  - percent of launches with non-empty timeline
  - percent with image license metadata

**G. Rollout + rollback**

- [ ] Roll out in order:
  1. write fields to `launches` only (no public cache)
  2. propagate to public cache
  3. enable UI display
- [ ] Rollback:
  - disable UI flag first
  - stop propagating to public cache if needed
  - keep stored data (harmless) unless we decide to prune later

### Exit criteria (Phase 2 complete)

- Launch timelines are ingested and displayed reliably.
- No meaningful increase in ingestion failures or cache refresh latency.
- Public cache contains the new fields and detail pages use them (no live LL2 calls required).

## Phase 3 — LL2 Event ingestion (selective, launch-linked)

### Objective

Ingest LL2 **Event** details (from `/events/{id}`) where we have strong linkage to launches, without blowing the rate limit.

Primary source of linkage in our system today:
- SNAPI ingestion produces `snapi_item_events(ll2_event_id)` (event IDs only).

### Deliverables

- New storage for event details + launch relationships (minimal viable schema).
- A rate-limited ingestion step that:
  - fetches missing/stale events referenced by SNAPI (and optionally other sources)
  - updates joins so we can show “related events” on a launch detail page

### Work plan

**A. Schema design (minimal + forward-compatible)**

- [ ] Add tables (names TBD):
  - `ll2_events` (PK: `ll2_event_id` int)
    - store: name/description/type/date/date_precision/location/image/info_urls/vid_urls/updates/webcast_live/last_updated
    - store: raw JSONB snapshot for forward compatibility
  - `ll2_event_launches` (join)
    - `ll2_event_id` (FK)
    - `launch_id` (FK to `public.launches.id`) or `ll2_launch_uuid` (FK to `launches.ll2_launch_uuid`)
  - (optional) `ll2_event_agencies` / `ll2_event_astronauts` later if needed; not required for initial launch linkage.

**B. Ingestion logic**

- [ ] Source queue:
  - Start from `snapi_item_events.ll2_event_id`.
  - Determine which IDs are not present in `ll2_events` or have stale `last_updated`.
- [ ] Fetch strategy:
  - Use `/events/{id}/?mode=detailed` (or the default detailed endpoint) and store the payload.
  - Respect rate limit strictly:
    - cap events fetched per ingestion run (e.g. 1–2 events/run)
    - store cursor/watermark in `system_settings` to continue next run
- [ ] Join creation:
  - For each fetched event, read its `launches[]` list and link to our launches by `ll2_launch_uuid`.

**C. UI (optional in Phase 3, but recommended)**

- [ ] Launch detail page:
  - show “Related events” (event name, type, date, link-outs) if any.
  - keep this behind a feature flag initially.

**D. Validation**

- [ ] Confirm the join works for known SNAPI-linked events.
- [ ] Ensure no orphan event links (event references a launch we don’t store because of US-only filter).
  - In such cases: keep the event stored but only link what matches our `launches`.

**E. Rollout + rollback**

- [ ] Roll out event ingestion disabled by default; enable after Phase 2 is stable.
- [ ] Rollback = disable the ingestion step; keep stored events.

### Exit criteria (Phase 3 complete)

- We can reliably fetch and store LL2 event details for SNAPI-linked event IDs.
- Launch detail pages can show related events where applicable (optional, feature-flagged).

---

## Safety checklist (applies to all phases)

- [ ] Feature flag every user-visible change; default off until data is present.
- [ ] Log ingestion stats to `ingestion_runs.stats` for quick diagnosis.
- [ ] Never exceed configured LL2 rate limits: enforce caps per run and reuse our atomic limiter.
- [ ] Prefer ingesting embedded objects from `launches/?mode=detailed` rather than extra endpoint calls.
- [ ] Keep changes reversible (configurable base URL, step-level toggles in ingestion-cycle).

---

## Open questions / decisions to lock before implementation

1) **Provider country mapping (2.3 country object):** store `alpha_3_code` (e.g. `USA`) vs `alpha_2_code` (e.g. `US`)?
2) **Timeline parsing:** do we want to store a computed “next milestone” for fast UI, or compute on the fly?
3) **Event schema:** store minimal typed columns + raw JSONB, or JSONB only initially?
4) **Historical remediation:** accept partial historical fields vs. run a one-time, time-bounded repair job.
